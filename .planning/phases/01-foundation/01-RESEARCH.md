## Summary

Phase 1 is a pure-Rust systems daemon. The core technical challenge is bridging two Linux kernel subsystems — the evdev input subsystem for reading physical key events AND for injecting synthetic key events — and the ALSA/PipeWire audio subsystem via CPAL. Both subsystems require elevated permissions via group membership (`input` for `/dev/input/*` read access and `/dev/uinput` write access), making error handling and permission-checking critical to the user experience.

The `evdev` crate (v0.13.2) is the single-crate solution for both PTT detection (reading from `/dev/input/event*`) and key injection (creating a `VirtualDevice` via `/dev/uinput`). This avoids introducing a separate `uinput` crate. CPAL (v0.17.3) handles audio capture cross-driver. The PTT "warm stream + gate" pattern uses a lock-free SPSC ring buffer (`ringbuf` 0.4.8) between the CPAL callback (producer) and the downstream pipeline consumer, ensuring the audio callback never blocks.

A critical toolchain finding: **Rust is not installed on the development machine**. The first plan must install `rustup` before any Rust compilation. `cargo-about` (v0.8.4) generates `LICENSES.md`; the project's own AGPL-3.0 is excluded via the `private` config or a filtered Handlebars template. `serde_yaml` is unmaintained and deprecated — use `serde_yaml_ng` (v0.10, MIT) as the drop-in replacement. Tokio's `signal` module handles SIGTERM/SIGINT gracefully.

**Primary recommendation:** Use the `evdev` crate exclusively for both input reading and uinput injection. Do not add the `uinput` crate — it is less maintained and doesn't offer uinput support that evdev doesn't already provide.

## Architectural Responsibility Map

| Capability | Primary Tier | Secondary Tier | Rationale |
|------------|-------------|----------------|-----------|
| Audio capture | Audio RT thread (CPAL-managed) | — | CPAL spawns its own high-priority callback thread; the callback writes to ring buffer only |
| PTT key detection | Dedicated OS thread | — | Blocking read loop on evdev fd; cannot be on Tokio executor (blocks) |
| Key injection / macro execution | Dedicated OS thread | — | D-07; blocking sleep loops for dwell/gap timing |
| PTT → audio gate signal | Lock-free atomic flag | — | `AtomicBool` between PTT thread (writer) and CPAL callback (reader); no allocations |
| Config loading | Main thread (startup) | — | Synchronous; happens once before any threads spawn |
| Signal handling (SIGTERM/SIGINT) | Tokio async task | — | `tokio::signal` is async-native; graceful shutdown via `CancellationToken` |
| Daemon lifecycle / supervision | Tokio runtime | — | Coordinates all other threads via channels; joins on shutdown |

### Pattern 1: PTT AtomicBool Gate in CPAL Callback

```rust
// Source: project pattern; CPAL docs confirm callback must not allocate
use std::sync::atomic::{AtomicBool, Ordering};
use std::sync::Arc;
use ringbuf::{HeapRb, Prod};

let ptt_active = Arc::new(AtomicBool::new(false));
let ptt_clone = Arc::clone(&ptt_active);

// HeapRb pre-allocates capacity at construction; callback never allocates
let rb = HeapRb::<f32>::new(16000 * 2); // 2 seconds at 16 kHz
let (mut producer, consumer) = rb.split();

let stream = device.build_input_stream(
    &config,
    move |data: &[f32], _: &cpal::InputCallbackInfo| {
        if ptt_clone.load(Ordering::Relaxed) {
            // Push samples to ring buffer — no allocation, no blocking
            let _ = producer.push_slice(data); // drops if buffer full; acceptable
        }
        // If PTT not held: samples silently discarded
    },
    |err| eprintln!("audio error: {err}"),
    None,
)?;
stream.play()?;
```

### Pattern 2: evdev Device Enumeration for PTT Key

```rust
// Source: evdev 0.13.2 docs.rs; evdev::enumerate() function
use evdev::{enumerate, KeyCode};

fn find_ptt_device(target_key: KeyCode) -> Option<evdev::Device> {
    for (_path, device) in enumerate() {
        if device.supported_keys()
            .map_or(false, |keys| keys.contains(target_key))
        {
            return Some(device);
        }
    }
    None
}

// PTT thread — blocking read loop, no EVIOCGRAB per D-10
fn ptt_thread(mut device: evdev::Device, key: KeyCode, ptt_active: Arc<AtomicBool>) {
    loop {
        for event in device.fetch_events().unwrap() {
            if let EventSummary::Key(_, k, value) = event.destructure() {
                if k == key {
                    ptt_active.store(value == 1, Ordering::Relaxed);
                    // D-12: only log at TRACE/DEBUG level
                    tracing::trace!("PTT key {k:?} value={value}");
                }
            }
        }
    }
}
```

### Pattern 3: evdev uinput VirtualDevice (Keyboard-Only)

```rust
// Source: evdev 0.13.2 docs.rs uinput module
use evdev::{uinput::VirtualDeviceBuilder, AttributeSet, KeyCode};

fn create_virtual_keyboard(supported_keys: &[KeyCode]) -> std::io::Result<evdev::uinput::VirtualDevice> {
    let mut keys = AttributeSet::<KeyCode>::new();
    for k in supported_keys {
        keys.insert(*k);
    }
    VirtualDeviceBuilder::new()?
        .name("hd-linux-voice")
        .with_keys(&keys)?
        .build()
}

// Injection thread: key-hold (D-07, MCRO-02)
fn emit_key_hold(device: &mut VirtualDevice, key: KeyCode, dwell_ms: u64) -> std::io::Result<()> {
    use evdev::InputEvent;
    device.emit(&[InputEvent::new(evdev::EventType::KEY.0, key.0, 1)])?;
    std::thread::sleep(Duration::from_millis(dwell_ms));
    device.emit(&[InputEvent::new(evdev::EventType::KEY.0, key.0, 0)])?;
    Ok(())
}
```

### Anti-Patterns to Avoid

- **Calling `EVIOCGRAB` / `device.grab()`:** D-10 explicitly prohibits exclusive grab. The game must still receive PTT key events. Never call `device.grab()`.
- **Allocating in the CPAL callback:** Any heap allocation (Vec, Box, String) in the audio callback risks priority inversion and dropouts. Pre-allocate the ring buffer; use only fixed-size stack operations inside the callback.
- **Running PTT read loop on Tokio executor:** `fetch_events()` blocks the calling thread. Spawn it as a `std::thread::spawn` (dedicated OS thread), not `tokio::task::spawn` or `spawn_blocking`.
- **Running injection sleeps on Tokio executor:** Macro timing uses `std::thread::sleep` on the injection thread. Never `tokio::time::sleep` inside an OS-thread loop.
- **Using serde_yaml 0.9:** The crate is deprecated (March 2024) with an unresolved high-severity libyaml CVE. Use `serde_yaml_ng` instead.
- **Using the separate `uinput` crate:** It is less maintained than `evdev::uinput`. Use the `uinput` module inside the `evdev` crate.
- **Creating any display surface:** Do not import `winit`, `xcb`, `wayland-client`, or any compositor library. The daemon must never open a Wayland surface or X11 window (UI-01).

## Common Pitfalls

### Pitfall 1: CPAL Device Doesn't Support Mono 16 kHz f32

**What goes wrong:** `build_input_stream` returns `BuildStreamError::StreamConfigNotSupported`. Many PipeWire/PulseAudio devices report a default format of stereo 48 kHz. Passing `channels: 1, sample_rate: 16000` directly without checking may fail.

**Why it happens:** CPAL 0.17 allows constructing `StreamConfig` manually, but the backend validates it against the device. Most Linux sound servers virtualize sample rate and channel count, but not all devices advertise mono capture as a supported config.

**How to avoid:** Query `device.supported_input_configs()` first. Accept the first supported config that covers mono (or stereo if mono isn't available) and the 16000–48000 Hz range. If stereo is returned, downsample to mono inside the callback by averaging left and right channels. If sample rate isn't 16 kHz, resample in the pipeline (Phase 2 can add `rubato` for resampling).

**Warning signs:** `BuildStreamError::StreamConfigNotSupported` in logs during device initialization.

### Pitfall 2: /dev/input/event* Permission Denied for Non-Root

**What goes wrong:** `evdev::enumerate()` silently skips devices it can't open, returning an empty iterator. The PTT thread finds no devices and fails with D-11 behavior.

**Why it happens:** `/dev/input/event*` nodes are owned by group `input` (on Arch/CachyOS). A user not in the `input` group sees `EACCES` on every device open, and `enumerate()` silently skips them (documented behavior: "won't bubble up errors").

**How to avoid:** Before scanning, check if any `/dev/input/event*` node is readable. If none are, fail with an actionable error: `"Cannot read /dev/input — add yourself to the 'input' group: sudo usermod -aG input $USER"`.

**Warning signs:** `find_ptt_device()` returns `None` despite the physical device being connected.

### Pitfall 3: systemd v258 uinput Group Issue (2025)

**What goes wrong:** After systemd v258, the udev rule `GROUP="uinput"` stops working because systemd now ignores non-system groups. Even with the user in the `uinput` group, `/dev/uinput` remains inaccessible.

**Why it happens:** systemd v258 introduced stricter group handling in udev rules. The `uinput` group must be created as a system group (`groupadd -r uinput`) to be recognized. On many existing installations, it was created as a normal group.

**How to avoid:** Prefer `GROUP="input"` in udev rules (the `input` group already exists as a system group on all major distros). The error message (D-15) should recommend `usermod -aG input $USER` rather than `uinput`.

**Warning signs:** User is in `uinput` group per `groups` command, but `/dev/uinput` still returns permission denied.

### Pitfall 4: Tokio `spawn_blocking` for Long-Lived Threads

**What goes wrong:** Using `tokio::task::spawn_blocking` for the PTT read loop or injection loop causes the Tokio blocking thread pool to saturate. Unlike `std::thread::spawn`, `spawn_blocking` is intended for short-duration blocking tasks.

**Why it happens:** `spawn_blocking` threads are drawn from a pool capped at 512 threads; long-running occupants starve other blocking tasks. Worse, a 2025 bug report documented mysterious failures when mixing `spawn_blocking` with certain OS operations.

**How to avoid:** Always use `std::thread::spawn` for the PTT thread and injection thread. Pass a `CancellationToken` or `AtomicBool` shutdown flag so threads can self-terminate on daemon shutdown.

### Pitfall 5: serde_yaml Security Advisory

**What goes wrong:** Using `serde_yaml = "0.9"` pulls in a transitive dependency on `libyaml` with an unresolved high-severity CVE, potentially flagged by `cargo audit` or CI security scanning.

**Why it happens:** The original `serde_yaml` maintainer archived the repo in March 2024 without resolving the `libyaml` advisory.

**How to avoid:** Use `serde_yaml_ng = "0.10"` — it is a drop-in API replacement and is actively maintained.

### Pitfall 6: evdev VirtualDevice SYN_REPORT Timing

**What goes wrong:** Injected key events are ignored or garbled by the receiving application.

**Why it happens:** The evdev protocol requires every batch of input events to be terminated with a `SYN_REPORT` event. CPAL analogy: like flushing a buffer. The `VirtualDevice::emit()` method sends events as-is; if `SYN_REPORT` is missing, the kernel may queue them indefinitely.

**How to avoid:** The `evdev` crate's `VirtualDevice::emit()` automatically appends `SYN_REPORT` after the provided slice. Do NOT manually append `SYN_REPORT` — doing so double-sends it. Verify behavior against the evdev source at docs.rs/evdev/uinput.rs.

### CPAL Explicit StreamConfig with Fallback

```rust
// Source: CPAL 0.17.3 docs.rs — StreamConfig fields
use cpal::{StreamConfig, BufferSize, SampleFormat, SampleRate};
use cpal::traits::{DeviceTrait, HostTrait};

fn build_audio_config(device: &cpal::Device) -> anyhow::Result<StreamConfig> {
    // Try explicit mono 16kHz first
    let explicit = StreamConfig {
        channels: 1,
        sample_rate: 16000,
        buffer_size: BufferSize::Default,
    };
    // Probe if supported (build then immediately drop to check)
    // Fallback: pick first supported config in 16k-48k range
    let supported = device
        .supported_input_configs()?
        .filter(|c| c.min_sample_rate().0 <= 16000 && c.max_sample_rate().0 >= 16000)
        .next();
    match supported {
        Some(c) => Ok(c.with_sample_rate(SampleRate(16000)).into()),
        None => Ok(explicit), // let CPAL fail descriptively at build_input_stream
    }
}
```

async fn main() -> anyhow::Result<()> {
    let token = CancellationToken::new();
    let mut sigterm = signal(SignalKind::terminate())?;
    let mut sigint  = signal(SignalKind::interrupt())?;

    // Spawn daemon tasks, passing token.clone() to each

    tokio::select! {
        _ = sigterm.recv() => tracing::info!("SIGTERM received, shutting down"),
        _ = sigint.recv()  => tracing::info!("SIGINT received, shutting down"),
    }
    token.cancel();
    // Join OS threads before exit (via JoinHandle or channel drain)
    Ok(())
}

## Open Questions (RESOLVED)

1. **Does the target CachyOS machine have the `input` group and is it a system group?**
   - What we know: Arch Linux / CachyOS default installs include `input` as a system group
   - What's unclear: Whether the development user is already a member
   - **RESOLVED:** Plan 03 Task 2 adds a `check_input_readable()` preflight that validates `/dev/input` access at startup and exits with an actionable error + `usermod -aG input $USER` instruction if the user is not in the `input` group.

2. **Does CPAL successfully capture mono 16 kHz on PipeWire without explicit resampling?**
   - What we know: PipeWire virtualizes sample rate; most configurations report support
   - What's unclear: Whether PipeWire returns the requested rate or silently returns a different rate
   - **RESOLVED:** Plan 03 Task 1 calls `build_audio_config()` which logs the actual negotiated `StreamConfig` (sample rate + channel count) at INFO level at startup. If the negotiated rate differs from 16 kHz, the discrepancy is visible in the log; a graceful fallback or warning is emitted rather than a silent mismatch.

3. **Which exact key codes should the Phase 1 test macro support?**
   - What we know: VirtualDeviceBuilder requires declaring all supported keys upfront
   - What's unclear: Whether Phase 1 should declare all possible keys or just a test subset
   - **RESOLVED:** Plan 04 Task 1 defines a `VIRTUAL_KEYBOARD_KEYS` const covering arrow keys, WASD, number row 0-9, F1-F12, Ctrl/Alt/Shift/Tab/Escape, and common game keys. This is a broad game-oriented set that covers all HD2 stratagem inputs without declaring every possible key.
